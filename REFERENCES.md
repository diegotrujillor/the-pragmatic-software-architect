# Referencias canónicas

Fuente primaria por volumen. Cada capítulo debe apoyarse en estos libros
(y en documentación oficial cuando aplique) antes de traducir el concepto a
lenguaje práctico y al sistema de ejemplo del libro.

| Volumen | Libro | Autor | Editorial | Rol en el volumen |
|---|---|---|---|---|
| 01 — Clean Architecture | *Clean Architecture* | Robert C. Martin | Pearson | Dependency Rule, capas, use cases |
| 02 — Domain-Driven Design | *Domain-Driven Design* | Eric Evans | Addison-Wesley | Aggregates, Value Objects, Bounded Contexts, Domain Events |
| 03 — Design Patterns | *Head First Design Patterns*, 2nd ed | Freeman & Robson | O'Reilly | Patrones clásicos GoF explicados con ejemplos |
| 04 — Testing | *Unit Testing: Principles, Practices, and Patterns* | Vladimir Khorikov | Manning | Qué testear, mocks vs. stubs, tests como red de seguridad |
| 05 — Microservices | *Building Microservices*, 2nd ed | Sam Newman | O'Reilly | Boundaries de servicio, trade-offs, cuándo NO usarlos |

## Cómo se usan

1. Extraer la idea central del libro (nivel 2 — Ingeniería).
2. Contrastarla con cómo decidiría un senior con 10 años (nivel 3).
3. Aterrizarla en el sistema de ejemplo: Merchant → Warehouse → Order →
   Package → Delivery.
4. Señalar dónde la IA cambió el cálculo costo/beneficio del patrón (sección
   🤖 de cada capítulo).

Referencias adicionales (documentación oficial, no libros) se citan inline
en cada capítulo, no aquí — este archivo es solo el mapa libro↔volumen.
